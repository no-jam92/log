효율적인 상태 관리를 위한 아키텍처 패턴

2024. 10. 24
12 min read
ARCHITECTURE REACT

프론트엔드 애플리케이션의 규모가 커짐에 따라 상태 관리는 가장 복잡하고 다루기 어려운 문제가 되었습니다. 이 글에서는 단방향 데이터 흐름을 유지하면서도 도메인 로직을 효과적으로 분리하는 방법에 대해 깊이 있게 탐구해 봅니다.

문제의 시작

우리는 종종 컴포넌트 내부에 상태와 비즈니스 로직을 혼재해서 작성하곤 합니다. 초기에는 개발 속도가 빠르고 직관적이기 때문에 이 방식이 선호됩니다. 하지만 요구사항이 복잡해지고 여러 컴포넌트에서 같은 상태를 공유해야 하는 순간, 프롭스 드릴링(Props Drilling)과 예기치 않은 사이드 이펙트의 늪에 빠지게 됩니다.

"복잡성은 소프트웨어 개발의 근본적인 한계다. 우리는 복잡성을 완전히 제거할 수 없지만, 아키텍처를 통해 그것을 국소화하고 격리할 수는 있다."

이러한 문제를 해결하기 위해 도입된 다양한 전역 상태 관리 라이브러리들(Redux, MobX, Zustand 등)은 각자의 철학을 가지고 있습니다. 그러나 도구의 선택보다 중요한 것은 상태를 '어떻게' 분류하고 설계할 것인가에 대한 원칙입니다.

상태의 분류

효율적인 관리를 위해 우리는 애플리케이션의 상태를 크게 세 가지로 분류해야 합니다.

  • UI 상태: 모달의 열림/닫힘, 드롭다운의 활성화 등 특정 컴포넌트나 뷰의 렌더링에만 관여하는 일회성 상태입니다.
  • 서버 상태: API 요청을 통해 가져온 데이터로, 클라이언트가 아닌 서버가 원본(Source of Truth)을 소유합니다. 비동기적이며 캐싱이 필요합니다.
  • 클라이언트 상태: 브라우저 세션 동안 유지되어야 하는 사용자 설정, 폼 입력 데이터, 장바구니 등 클라이언트가 소유권을 가지는 전역 상태입니다.
store/useAuthStore.ts 

import { create } from 'zustand';

interface AuthState {
  user: User | null;
  isAuthenticated: boolean;
  login: (user: User) => void;
  logout: () => void;
}

export const useAuthStore = create<AuthState>((set) => ({
  user: null,
  isAuthenticated: false,
  login: (user) => set({ user, isAuthenticated: true }),
  logout: () => set({ user: null, isAuthenticated: false }),
}));

위 코드는 Zustand를 활용한 간단한 클라이언트 상태 관리의 예시입니다. 보일러플레이트가 적고 직관적인 API 덕분에 최근 많은 주목을 받고 있습니다. 중요한 점은 도메인별로 스토어를 분리하여 관심사를 명확히 했다는 것입니다.

결론

완벽한 아키텍처란 존재하지 않습니다. 프로젝트의 규모, 팀의 숙련도, 그리고 비즈니스 요구사항에 따라 적절한 도구와 패턴을 선택하는 안목이 필요할 뿐입니다. 다만, 상태의 소유권과 생명주기를 명확히 정의하는 것에서부터 건강한 코드베이스가 시작된다는 점을 잊지 말아야 합니다.
