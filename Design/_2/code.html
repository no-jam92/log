리액트 컴포넌트 설계의 최소주의

조용한 개발자
• 2023. 10. 24. • 읽기 7분

복잡한 프론트엔드 애플리케이션을 개발하다 보면 컴포넌트가 점점 비대해지는 것을 목격하게 됩니다. 상태 관리는 얽히고, 렌더링 최적화는 산으로 가며, 결국 하나의 파일을 열었을 때 수백 줄의 코드가 우리를 압도합니다. 이 글에서는 최소주의적 접근을 통해 어떻게 리액트 컴포넌트를 본연의 목적에 맞게 작고 단단하게 유지할 수 있는지 탐구해 봅니다.

왜 이 글을 쓰게 되었나

최근 진행했던 대규모 리팩토링 프로젝트에서 가장 큰 병목은 '모든 것을 다 하는' 거대 컴포넌트들이었습니다. 데이터 페칭, 복잡한 비즈니스 로직, 세밀한 UI 조작이 하나의 함수 안에 혼재되어 있었죠. 우리는 코드를 읽는 시간을 줄이고, 변경에 유연하게 대처할 수 있는 구조가 필요했습니다.

"완벽함이란 더 이상 보탤 것이 없을 때가 아니라, 더 이상 뺄 것이 없을 때 달성된다."
— 앙투안 드 생텍쥐페리

컴포넌트의 책임 분리

최소주의 설계의 첫 걸음은 관심사의 분리(Separation of Concerns)입니다. UI 컴포넌트는 오직 어떻게 보일 것인가에 집중해야 합니다. 데이터가 어디서 오는지, 상태가 어떻게 변하는지 알 필요가 없습니다.

UserProfile.tsx 
// Bad: 데이터 페칭과 UI 렌더링이 결합됨
function UserProfile({ userId }) {
  const { data, loading } = useQuery(`/api/users/${userId}`);
  
  if (loading) return <Spinner />;
  
  return (
    <div className="profile-card">
      <h1>{data.name}</h1>
      <p>{data.bio}</p>
    </div>
  );
}

// Good: UI 렌더링에만 집중
function UserProfileCard({ name, bio }) {
  return (
    <div className="profile-card">
      <h1>{name}</h1>
      <p>{bio}</p>
    </div>
  );
}

위 예시처럼 데이터를 주입받는 단순한 프레젠테이셔널 컴포넌트를 만들면, 테스트하기 쉽고 재사용성이 높아집니다. 디자인 시스템의 핵심은 이러한 순수 컴포넌트들의 집합을 구축하는 것입니다.
